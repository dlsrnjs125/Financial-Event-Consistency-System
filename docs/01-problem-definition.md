# 금융 이벤트 정합성 시스템 - 문제 정의

## 프로젝트 한 줄 소개

외부 금융 시스템에서 재시도·타임아웃·네트워크 지연으로 중복 수신되는 거래 이벤트를 **Idempotency Key, PostgreSQL Transaction, Redis, 상태 머신**으로 안전하게 처리하는 정합성 검증 시스템

---

## 해결하려는 문제

### 핵심 문제 상황

외부 금융 시스템(은행, 결제사, 증권사)은 우리 API의 응답을 받지 못하면 동일한 거래 이벤트를 다시 보낼 수 있습니다.

```
1️⃣ 외부 은행 시스템 → 입금 이벤트 전송 (BANK-20260527-000001)
2️⃣ 우리 API → DB에 입금 반영, 잔액 증가
3️⃣ 응답 전송 중 네트워크 타임아웃
4️⃣ 외부 시스템 → 응답을 받지 못해 같은 이벤트 재전송
5️⃣ 우리 API → 중복 여부 확인 없이 다시 처리
6️⃣ 결과 → 사용자 잔액이 2배로 증가 (정합성 깨짐)
```

### 실제 발생하는 금융 문제들

| 문제 | 예시 | 피해 |
|-----|-----|-----|
| 네트워크 지연 | API 응답은 성공했지만 외부 시스템이 타임아웃으로 판단 | 이벤트 재전송 |
| 동시 요청 | 동일한 Idempotency Key로 여러 요청 동시 도착 | 중복 처리 위험 |
| 상태 꼬임 | CANCEL 이벤트가 DEPOSIT 이벤트보다 먼저 들어옴 | 거래 상태 깨짐 |
| 배포 중 유실 | Blue-Green 배포 중 요청이 유실되고 재시도됨 | 중복 거래 반영 |
| 외부 시스템 장애 복구 | 외부 시스템 장애 후 미처리로 판단한 이벤트 재전송 | 며칠 후 중복 반영 |

---

## 해결하지 않으면 생기는 피해

### 👥 사용자 피해
- **입금이 두 번 반영됨** → 잔액 오류, 출금 불가
- **결제가 중복 승인됨** → 중복 결제, 환불 대기
- **취소된 거래가 다시 완료됨** → 거래 상태 혼란
- **잔액이 실제와 다르게 표시됨** → 신뢰도 하락

### 💼 회사 피해
- **금전 손실** → 중복 입금만큼의 손실 발생
- **정산 불일치** → 외부 시스템과의 정산 데이터 맞지 않음
- **감사 대응** → 금융감독청 검사 시 설명 곤란
- **민원 폭증** → 고객 센터 부담 증가
- **브랜드 손상** → 금융시스템 신뢰도 하락

### 🔧 운영자 피해
- **원인 추적 어려움** → 같은 이벤트가 몇 번 처리됐는지 알 수 없음
- **수동 보정 증가** → 데이터베이스 직접 수정 필요
- **장애 대응 시간 증가** → 근본 원인 파악에 오래 걸림
- **운영 부담 증가** → 야간/주말 대응 빈도 증가

## 이 시스템이 보장할 것

### 핵심 보장 사항

1. **동일 이벤트는 여러 번 들어와도 한 번만 처리한다**
   - `external_event_id = BANK-001`이 100번 들어와도
   - `transaction_event` 테이블에는 1행만 생성
   - 계좌 잔액도 1회만 변경

2. **동일 Idempotency Key 요청은 항상 같은 결과를 반환한다**
   - 첫 요청: 신규 처리
   - 재요청: 저장된 응답 그대로 반환

3. **같은 Idempotency Key로 다른 요청이 오면 즉시 거부한다**
   - 같은 Key, 다른 Amount → 409 Conflict
   - 같은 Key, 다른 Account → 409 Conflict

4. **잘못된 상태 전이는 차단한다**
   - COMPLETED → PROCESSING (불가)
   - FAILED → COMPLETED (불가)
   - 잘못된 전이 시도 → 예외 발생

5. **Redis 장애가 발생해도 최종 정합성은 PostgreSQL에서 보장한다**
   - Redis Down 상황에서도 중복 거래 반영 방지
   - PostgreSQL의 UNIQUE Constraint가 최종 방어선

6. **배포 전 정합성 테스트를 통과하지 못하면 배포하지 않는다**
   - 중복 이벤트 테스트 실패 → 배포 차단
   - 잘못된 상태 전이 코드 → 배포 차단

---

## 프로젝트의 핵심 메시지

> 💡 금융 이벤트 처리 시스템에서 중요한 것은 빠른 응답만이 아니다.
> 
> 동일한 이벤트가 여러 번 들어와도 결과가 한 번만 반영되고,
> 
> 잘못된 순서의 이벤트가 들어와도 거래 상태가 깨지지 않는 것이 핵심이다.

---

## 배운 점 및 설계 원칙

### 가장 중요한 설계 원칙

```
❌ 틀린 접근: Redis Lock만으로 정합성을 보장한다
✅ 올바른 접근: PostgreSQL을 최종 정합성의 기준으로 두고,
                Redis는 성능 최적화 계층으로만 사용한다
```

**왜냐하면:**
- Redis는 메모리 기반이므로 언제든 장애 가능
- Redis가 죽으면 Lock도 Cache도 사라짐
- 따라서 PostgreSQL의 UNIQUE Constraint와 Transaction으로 최종 방어
