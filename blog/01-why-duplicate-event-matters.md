# 1편. 왜 금융 이벤트 처리 시스템에서 중복 처리가 중요한가?

## 들어가며

금융 이벤트 처리에서 가장 치명적인 문제는 **속도가 느린 것이 아니라 중복 처리**입니다.

외부 시스템이 우리 API의 응답을 받지 못하면, 동일 이벤트를 재전송합니다.
이때 우리 시스템이 중복 여부를 확인하지 않으면:
- 입금이 2번 반영
- 결제가 중복 승인
- 사용자 잔액이 2배로 증가

---

## 실제 발생하는 중복 이벤트 상황

### 상황 1: 네트워크 타임아웃
```
[은행] DEPOSIT 이벤트 전송
        ↓
[우리 API] 처리 성공 → DB에 잔액 +10,000 반영
        ↓
[우리 API] 응답 전송 중... 네트워크 끊김 ⚠️
        ↓
[은행] 타임아웃으로 판단 → 같은 이벤트 재전송
        ↓
[우리 API] 중복 확인 안 함 → 잔액 +10,000 또 증가 ❌
결과: 실제로는 10,000원이었는데 20,000원으로 표시됨
```

### 상황 2: 동시 요청
```
같은 Idempotency Key로 여러 서버에서 동시 요청
        ↓
Race condition으로 각 서버가 각각 처리
        ↓
결과: 같은 거래가 여러 번 DB에 반영됨
```

### 상황 3: 배포 중 요청 유실
```
Blue-Green 배포 중 트래픽 전환
        ↓
일부 요청이 응답을 받지 못함
        ↓
외부 시스템 자동 재시도
        ↓
새 버전에서도 처리 → 중복 반영
```

---

## 해결책 검토

### ❌ 단순 접근: 외부 이벤트 ID로 조회 후 존재하면 무시
```
이미 처리했는가?
  → YES: 반환
  → NO: 처리
```

**문제점**:
- 조회 시간과 INSERT 사이의 Race Condition
- 동시 요청 2개가 모두 "미처리"로 판단하고 INSERT 실행 가능

### ❌ Redis Lock만 믿기
```
Redis Lock 획득 → 처리 → Lock 해제
```

**문제점**:
- Redis 장애 시 Lock을 못 걸었는데 이미 처리된 거래 중복 반영
- Redis 재시작 후 메모리 초기화 → Lock 사라짐

### ✅ PostgreSQL Unique Constraint + Transaction 사용
```
BEGIN TRANSACTION
  ↓
Idempotency Key 확인
  ↓
UNIQUE Constraint 있는 테이블에 INSERT
  ↓
실패 → 중복 반영 방지 ✅
  → 성공하면 처리 진행
COMMIT
```

## 이 프로젝트의 검증 방식

### 검증 1: 동일 이벤트 100번 동시 요청
```
external_event_id = BANK-001인 이벤트를 100번 동시 요청
        ↓
확인:
- transaction_event 테이블: 1행만 존재? ✅
- ledger_entry 테이블: 1행만 존재? ✅
- account.balance: 1회만 증가? ✅
```

### 검증 2: 같은 Idempotency Key로 다른 금액 요청
```
Idempotency Key = "idem-001"로 {amount: 10000} 요청 성공
이후 Idempotency Key = "idem-001"로 {amount: 50000} 요청
        ↓
기대 결과:
- 409 Conflict 반환 ✅
- 거래는 {amount: 10000}만 반영 ✅
```

### 검증 3: Redis 장애 상황
```
Redis 다운 → Lock 사용 불가
        ↓
동일 이벤트 100번 요청
        ↓
PostgreSQL Unique Constraint로 중복 방지 ✅
```

## 핵심 메시지

> **금융 이벤트 처리에서 중요한 것은 빠른 응답이 아니라, 중복·재시도·장애 상황에서도 거래 결과가 한 번만 정확하게 반영되는 것이다.**
